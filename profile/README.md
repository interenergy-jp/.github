# Interenergy GitHub Organization

インターエナジーのソフトウェア、設計データおよび関連ドキュメントを管理するGitHub Organizationです。

## 基本方針

リポジトリは原則として **Private** で作成します。

## Claude Code スキルの利用

この Organization の定型操作（PR作成・Issue起票・リポジトリ作成など）は、
社内配布の Claude Code スキル集 [ie-agent-skills](http/ie-agent-skills) で自動化されています。
Claude Code 上で以下を一度実行すると利用できます（詳細は同リポジトリのREADME参照）。

```text
/plugin marketplace add interenergy-jp/ie-agent-skill
/plugin install gh-ops@ie-agent-skills
```

以降の各節では、手作業の手順とあわせてスキルでの操作
スキルを使わず手作業で運用しても問題ありません。

  * 組み込みファームウェア

  * その他

## リポジトリ名

原則として、次の形式を使用します。

```text
PJ番号上3文字（または２文字）-年度2桁-PJCode-内容
```

例： PJ番号　SAE-26001-00

```text
sae-26-repeater-fw
sae-26-repeater-gui
```

複数の構成要素がある場合は、それぞれの役割が分かる名前にしてください。

リポジトリ名には、原則として次の形式を使用します。

* 半角英数字
* 小文字
* 単語の区切りはハイフン
* 略称は社内で意味が共有されているものに限定
* IE以外の会社名、製品名を直接入れない

年度2桁は `repo-create` スキルが作成日から自動で付与します（4月始まりの年度）。

## ブランチ運用

通常の開発では、デフォルトブランチを直接変更せず、作業ブランチからPull Requestを作成します。
下記に具体例を示しますが、ブランチ名は自由に作成して
PRを挙げ、デフォルトブランチに取り込まれると自動で作業ブランチは削除されます。

**Claude Code を使う場合**は「PRを作って」と依頼するだけで、`pr-create` スキルが
以下の一連の手順（ブランチ作成→コミット→Push→PR作成）

作業ブランチ名の例：

```text
feature/作業名
edit/作業名
```

基本的な作業手順：

```bash
git switch master
git pull origin master
git switch -c feature/作業内容
```

変更後、作業ブランチをPushします。

```bash
git add .
git commit -m "変更内容"
git push -u origin feature/作業内容
```

GitHub上でPull Requestを作成したら、レビューおよび必要な確認を経てマージされます。

## デフォルトブランチ

既存プロジェクトでは、デフォルトブランチとして `master` を使用している場合があります。

新規プロジェクトで `main` を使用する場合は、プロジェクト内で表記を統一してください。

以下の操作は原則として禁止します（前2つはOrganizationのルールセットでも強制されています）。

* デフォルトブランチへの直接Push
* Force Push
* レビューを経ない変更のマージ
* 履歴を書き換える操作
* 管理者への確認なしでのデフォルトブランチ変更

## Pull Request

Pull Requestには、少なくとも以下を記載してください。
（`pr-create` スキルを使う場合は変更内容から自動で起

* 変更内容概要
* 動作確認結果
* 補足など

記載例：

```markdown

UART通信のタイムアウト処理を追加しました。
```

レビュワーは1名以上指定し、うち1名以上はマネージャー 。
作成者自身がマネージャーの場合は省略できます。`pr-create` スキルはこのルールに沿って

Claude Code では「Issueを立てて」と依頼すると `issue-

## レビュー

レビューでは、主に以下を確認します。

* 要求仕様を満たしているか
* 既存機能への影響がないか
* 可読性や保守性に問題がないか
* エラー処理が適切か
* セキュリティ上の問題がないか
* 必要な動作確認が行われているか
* 機密情報が誤って含まれていないか

指摘事項への対応が完了し、未解決の会話がないことを確認してからマージしてください

## コミット

コミットメッセージは、変更内容が分かるように記載して

良い例：

```text
UART受信タイムアウト処理を追加
ログイン失敗時のエラー表示を修正
データベース接続処理を共通化
セットアップ手順を更新
```

避ける例：

```text
修正
更新
test
work
一旦保存
```

可能な範囲で、1つのコミットには関連する変更だけを含めてください。

## 既存の社内Gitからの移行

社内Gitを `origin` としている既存リポジトリは、GitHubへの移行完了後に接続先を変更します。

```bash
git remote -v
# URLは対象のものに書き換えること
git remote set-url origin https://github.com/interenergy-jp/med-26-adee-fw.git
git fetch origin --prune
git branch -vv
```

未コミットの変更は、`origin` を変更しても失われません。

接続先の変更後に、通常どおりコミットしてGitHubへPushできます。

作業中のブランチを初めてGitHubへPushする場合は、以下を実行します。

```bash
git push -u origin ブランチ名
```

移行後は、旧社内GitへPushしないでください。
